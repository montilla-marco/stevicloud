Component Diagram — product-catalog

How is the ProductCatalog bounded context structured internally? Hexagonal Architecture (Ports & Adapters) — domain has zero framework imports.

Domain Layer — Zero Framework Imports
Product Aggregate Root

Enforces all product invariants. Lifecycle: DRAFT → PUBLISHED → DISCONTINUED. Registers domain events on each transition.

ProductId · ProductName · Category · ProductStatus Value Objects

Immutable, self-validating. Invalid states unrepresentable at the type level — no nulls escape domain construction.

ProductCreated · ProductPublished · ProductDiscontinued Domain Events

Past-tense, immutable facts. Carry eventId (UUID) for idempotency. Drained from aggregate by application layer after save.

DomainEvent Marker Interface

eventId(): String, occurredOn(): Instant. The only contract events must satisfy.

ProductRepository Port · «interface»

Driven port. save(Product), findById(ProductId), findAll(). Defined in domain — implemented in infrastructure.

Application Layer — Orchestration Only
CreateProductUseCase Command Handler

Calls Product.create(), repository.save(), then drains and publishes domain events via DomainEventPublisher port.

PublishProductUseCase Command Handler

Loads aggregate, calls product.publish() (throws if not DRAFT), saves, drains events.

GetProductUseCase Query Handler

Read-only path. Returns Product by ID or all products. No events raised.

DomainEventPublisher Port · «interface»

publish(DomainEvent). Driving port used by use cases. Stage 1: logging. Stage 2: Kafka.

Infrastructure — Driving Adapters (in)
ProductController Adapter · Spring MVC

REST endpoints — POST /api/v1/products, GET /api/v1/products/{id}, PUT /api/v1/products/{id}/publish. Translates HTTP ↔ Commands/Queries. No domain logic.

CreateProductRequest · ProductResponse DTOs · Spring MVC

Jackson-serialized transfer objects. Not part of the domain model. Response hides internal aggregate structure.

Infrastructure — Driven Adapters (out)
InMemoryProductRepository Adapter · Stage 1 → PostgresProductRepository (Stage 2)

Implements ProductRepository port using a ConcurrentHashMap. Zero-dependency, swappable without touching domain or use cases. Replaced by JPA adapter in Episode 2.

LoggingDomainEventPublisher Adapter · Stage 1 → KafkaDomainEventPublisher (Episode 2)

Implements DomainEventPublisher port using SLF4J. Events printed as structured JSON log lines. Replaced by Kafka adapter in Episode 2 without modifying any use case.

ApplicationConfig Spring @Configuration

Wires port interfaces to adapter implementations. The only place Spring is aware of which adapters are active. Constructor injection only.

Rule — domain/ imports nothing outside its own package. Verified by running tests without Spring context. Test strategy — ProductTest.java: plain JUnit, no mocks, sub-second. CreateProductUseCaseTest: use case + InMemory adapter, no Spring. Next — Episode 2 swaps InMemory→Postgres and Logging→Kafka adapters. Zero domain code changes.

Series

Episode Roadmap

Each Episode is one vertical slice — a bounded context fully implemented from domain model to REST adapter, production-quality from day one. [Ford-BEA] Ch.3.

E-01
ProductCatalog BC — DDD + Hexagonal
Product aggregate, 3 value objects, 3 domain events, 3 use cases, REST adapter, 11 tests. In-memory + log-based adapters.
✓ Done
E-02
Pricing BC — Kafka + Transactional Outbox
Strimzi Kafka on cluster. KafkaDomainEventPublisher replaces Logging adapter. PostgresProductRepository replaces InMemory. Pricing BC subscribes to ProductPublished. EIP: Idempotent Consumer, Dead Letter Queue.
Planned
E-03
Inventory BC — MongoDB + Polyglot Persistence
Warehouse & inventory management. Variable schema per SKU type (electronics→serial, food→expiry, clothing→size/color). Nested location documents (warehouse→aisle→shelf→bin). Stock movements as append-only events. Consumes ProductPublished to initialize InventoryItem. Polyglot persistence: MongoDB alongside the PostgreSQL services.
Planned
E-04
OMS — NestJS + Event-Sourced Order
Order Management BC in NestJS. Consumes ProductPublished + InventoryReserved. OrderPlaced / OrderConfirmed events. Saga / Process Manager pattern for multi-step order fulfillment.
Planned
E-05
Billing — Stripe Integration + Invoice BC
Billing BC in NestJS. Consumes OrderConfirmed. Stripe payment initiation (WireMock in tests). PDF invoice generation. Idempotent payment handling.
Planned
E-06
Platform Hardening — CI + Istio mTLS + Keycloak OIDC
GitHub Actions CI with nx affected on PR. Service mesh (Istio) with mutual TLS. Keycloak OIDC/JWT across all services. ArchUnit boundary enforcement. Testcontainers for PostgreSQL + Kafka + MongoDB in integration tests.
Planned
